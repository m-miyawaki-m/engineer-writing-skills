## 概要

出典: ログ設計ガイドライン / レベル: 中級

## 悪い例 1

## ユーザー登録API

```javascript
app.post('/api/users', async (req, res) => {
  console.log('リクエスト来た');
  console.log('body:', req.body);
  console.log('password:', req.body.password);
  
  const user = await UserService.create(req.body);
  console.log('ユーザー作成した:', user);
  console.log('ここ通った');
  
  res.json(user);
});
```

→ デバッグ用の `console.log` がそのまま本番に残っている。パスワードが平文でログに記録される。

## 良い例 1

## ユーザー登録API

```javascript
app.post('/api/users', async (req, res) => {
  const requestId = req.headers['x-request-id'] || uuid();
  
  logger.info('User registration started', {
    requestId,
    email: req.body.email,
    // password は絶対にログに出さない
  });
  
  try {
    const user = await UserService.create(req.body);
    logger.info('User registration completed', {
      requestId,
      userId: user.id,
      duration: Date.now() - startTime
    });
    res.json({ id: user.id, email: user.email });
  } catch (err) {
    logger.error('User registration failed', {
      requestId,
      errorCode: err.code,
      message: err.message
    });
    res.status(500).json({ error: 'Registration failed' });
  }
});
```

→ 構造化ログ、requestIdで追跡可能、機密情報なし、エラーハンドリング付き

## 解説 1

- **改善点**: 「console.log('リクエスト来た')」 -- デバッグ用の雑なメッセージ。本番では何の意味も持たない。リリース前に必ず削除すべき
- **改善点**: 「console.log('password:', req.body.password)」 -- 最も危険なパターン。パスワードを平文でログに出力している。ログ流出時に全ユーザーの認証情報が漏洩する
- **スキル**: 「requestId」 -- リクエストIDを付与することで、1つのリクエストに関連する全ログを追跡できる。障害調査の効率が格段に上がる
- **スキル**: 「logger.info / logger.error」 -- ログレベルを適切に分けることで、障害時にERRORだけフィルタして素早く原因特定できる
- **ノート**: 「duration: Date.now() - startTime」 -- 処理時間を記録しておくと、パフォーマンス劣化の検知に役立つ。SLI/SLO監視の基礎データになる

## 悪い例 2

## ログ設定

```javascript
// 全部ログに出す
const LOG_LEVEL = 'debug';

// ログはコンソールに出力
console.log(`[${new Date()}] ${message}`);
```

→ 本番でもDEBUGレベル全開。ログ量が膨大になりディスクを圧迫。タイムスタンプの形式もバラバラ。

## 良い例 2

## ログ設定

```javascript
const logger = createLogger({
  level: process.env.LOG_LEVEL || 'info',
  format: combine(
    timestamp({ format: 'YYYY-MM-DDTHH:mm:ss.SSSZ' }),
    json()
  ),
  transports: [
    new transports.Console(),
    new transports.File({
      filename: 'error.log',
      level: 'error',
      maxsize: 10485760,  // 10MB
      maxFiles: 5
    })
  ]
});
```

→ 環境変数でレベル制御、ISO 8601タイムスタンプ、JSON構造化、ログローテーション設定済み

## 解説 2

- **改善点**: 「LOG_LEVEL = 'debug'」 -- 本番でDEBUGレベルは厳禁。大量のログでディスクを圧迫し、重要なログが埋もれる。環境変数で制御すべき
- **スキル**: 「process.env.LOG_LEVEL || 'info'」 -- 環境変数でログレベルを制御。本番はINFO、障害調査時だけ一時的にDEBUGに切り替えられる
- **スキル**: 「timestamp + json()」 -- ISO 8601形式のタイムスタンプとJSON構造化で、ログ集約基盤（Datadog、CloudWatch）で検索・集計が容易になる
- **ノート**: 「maxsize + maxFiles」 -- ログローテーション設定。これがないとログファイルが無限に肥大化してディスク枯渇を引き起こす
