## 概要

本番リリース前にログ出力を点検するためのチェックリスト。開発用ログの残存・レベル設定・セキュリティ・構造化を網羅的に確認する

## 良い例

## ログ設計チェックリスト

### 1. デバッグログの除去
- [ ] `console.log` / `console.debug` / `print` 文がコードに残っていないか
- [ ] `// TODO: 後で消す` 付きのログ出力が残っていないか
- [ ] デバッガ用の一時変数・ダンプが残っていないか

### 2. ログレベルの適切さ
- [ ] ERROR: システムが正常に動作できない事象のみに使用
- [ ] WARN: 想定外だが処理は継続できる事象に使用
- [ ] INFO: 正常な業務イベント（開始/完了/重要な状態変化）に使用
- [ ] DEBUG: 本番では原則OFF、開発時のみ有効

### 3. セキュリティ
- [ ] パスワード・トークン・APIキーがログに含まれていないか
- [ ] 個人情報（PII）がマスキングされているか
- [ ] リクエストボディの全出力（`req.body`）をしていないか

### 4. 運用観点
- [ ] 構造化ログ（JSON形式）になっているか
- [ ] requestId / traceId でリクエスト追跡可能か
- [ ] タイムスタンプがISO 8601形式か
- [ ] ログローテーション設定があるか

## 悪い例

チェックリストなし。各開発者が自由にログを書き、レビューでもログの品質は確認しない
→ 本番ログが `console.log('debug:', data)` だらけ
→ 障害時に有用な情報がない

## 使いどころ

PR レビュー、リリース前確認、ログ設計方針の策定
