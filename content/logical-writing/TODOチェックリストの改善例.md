## 概要

出典: PRテンプレート / レベル: 初級

## 悪い例

- [ ] コード書いた
- [ ] テストした
- [ ] 確認した

## 良い例

## PR チェックリスト

### コード品質
- [ ] Lint / Formatter エラーなし（`npm run lint` 通過）
- [ ] 型チェック通過（`npx tsc --noEmit`）
- [ ] 不要な console.log / デバッグコード削除済み

### テスト
- [ ] 新規テスト追加済み（対象: ○○関数）
- [ ] 既存テスト全件通過（`npm test` 結果貼付）
- [ ] エッジケース確認済み（null, 空配列, 上限値）

### レビュー・ドキュメント
- [ ] セルフレビュー実施済み（diff 全行確認）
- [ ] 変更に関連する API ドキュメント更新済み
- [ ] CHANGELOG に追記済み（該当する場合）

## 解説

- **改善点**: 「コード書いた」 -- 完了条件が曖昧。「書いた」だけでは品質が担保されない
- **改善点**: 「テストした」 -- 何のテストをどう実行したか不明。具体的なコマンドと期待結果を書く
- **スキル**: 「npm run lint 通過」 -- 検証コマンドを明記することで、誰でも同じ基準で確認できる
- **スキル**: 「### コード品質」 -- カテゴリ分けすることで、チェック漏れを防ぎ、どの観点が不足しているか一目でわかる
