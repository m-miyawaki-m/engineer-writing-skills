## 説明

次工程に進む前に、前工程の成果物が完成しており次工程のインプットとして十分かを確認するレビューパターン

## 工程移行チェックリスト

### 要件定義 → 基本設計
- [ ] 全機能要件にIDが振られている
- [ ] 非機能要件（性能/セキュリティ/可用性）が定義済み
- [ ] ステークホルダーの承認が得られている
- [ ] 要件の優先度（Must/Should/Could）が決まっている

### 基本設計 → 概要設計
- [ ] アーキテクチャ図が作成されている
- [ ] 外部IF仕様（API一覧・形式）が定義済み
- [ ] DB設計（ER図）が完成している
- [ ] 全要件が基本設計のどこかに対応づいている

### 概要設計 → 詳細設計
- [ ] 全機能の処理フローが定義されている
- [ ] 画面レイアウトが確定している
- [ ] エラー処理方針が統一されている
- [ ] 機能テスト観点が洗い出されている

### 詳細設計 → 実装
- [ ] クラス/関数の入出力が定義されている
- [ ] 境界条件・例外処理が明記されている
- [ ] 単体テスト観点が洗い出されている
- [ ] コーディング規約が周知されている

## 悪い例

「だいたい設計できたので実装に入ります」（レビューなし、成果物の完成度が曖昧）

## 活用場面

プロジェクトマネジメント、品質ゲート、ウォーターフォール/ハイブリッド開発
